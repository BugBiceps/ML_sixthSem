Introduction of Compiler & Translation Model

A Compiler is a specialized system software program that translates a source program written in a high-level language (like C, C++, Java, Rust) into an equivalent target program written in a low-level language (like assembly language or machine code) without changing the functional meaning of the source code.

Why It Exists:

  • Hardware Gap — CPUs run on binary code. Compilers bridge the abstraction gap so humans can write logical code instead of manual assembly.
  • Portability — Separates source syntax from concrete CPU platforms by generating custom binaries for diverse targets.
  • Optimization — Optimizes operations globally (reducing register spills, memory latency, and instruction sizes).

Compiler vs. Interpreter:

Parameter Compiler Interpreter
Execution Speed Faster (pre-compiled to target machine instructions). Slower (translated line-by-line during runtime).
Output File Generates standalone intermediate target object/executable code. No intermediate output; translates code on-the-fly.
Memory Usage Higher (requires storing whole code representation and output). Lower (only stores instructions currently executing).
Error Detection Scans the entire source code and reports errors at the end. Stops execution at the very first error encountered.
Examples C, C++, Rust, Go, Swift, Haskell Python, JavaScript, Ruby, PHP, Perl
Types of Compilers

Compilers are engineered in different ways depending on compile-speed requirements, memory constraints, and target architectures:

Compiler Type Definition & Core Concept Key Feature / Application
Single-Pass Scans source code exactly once. Parses and emits code concurrently. Extremely fast compiler speed; minimal global optimization.
Multi-Pass Scans code multiple times, building intermediate representations. Allows extensive optimizations; clean division of phases.
Cross-Compiler Runs on one architecture (e.g., x86) but generates code for another (ARM). Critical for developing software on smartphones and embedded hardware.
Incremental Recompiles only modified subroutines rather than the whole codebase. Speeds up compilation in massive enterprise projects.
Just-In-Time (JIT) Compiles intermediate bytecode to machine instructions during execution. Used in Java Virtual Machines (JVM) and JavaScript V8.
Bootstrapping A compiler written in the same language it compiles (e.g., C compiler written in C). Requires compiling the initial compiler using a helper system.
Major Data Structures in Compilers

To store information and exchange it across phases, compilers use specialized data structures:

Data Structure Definition & Purpose Implementation Approach
Symbol Table Primary registry storing identifier properties (name, scope, type, memory offset). Hash Tables or Self-Balancing Trees for O(1) or O(log N) operations.
Abstract Syntax Tree (AST) A compact, hierarchical tree showing program structure, omitting punctuation/keywords. Tree nodes containing operator types and pointers to operands.
Parse Tree Graphical representation of grammar derivations; includes all terminal/non-terminals. Full tree showing 1:1 derivation path (Concrete Syntax Tree).
Literal Table Registry holding string constants and numeric literals. Array/Hash of constants to optimize allocation and offset mapping.
Front-End vs. Back-End & Analysis-Synthesis Model

The compilation pipeline is divided into two logical models: Analysis (Front-End) and Synthesis (Back-End).

Aspect Front-End (Analysis) Back-End (Synthesis)
Focus Source code structure, spelling, type matching, scope. Instruction selection, CPU register layout, performance optimization.
Platform Bound Machine-Independent (runs the same on any OS/CPU). Machine-Dependent (bound to target hardware/register set).
Inputs & Outputs Source Code -> Intermediate Representation (IR). Intermediate Representation (IR) -> Target Assembly/Binary.
Phases of a Compiler

A compiler consists of six distinct phases. The Symbol Table and Error Handler connect to all of them.

Step-by-Step Walkthrough: Compiling position = initial + rate * 60

  1. Lexical Analysis (Scanner): Reads character input and emits tokens. Assigns identifiers to entries in the Symbol Table.
    Token stream: <id, 1> <=> <id, 2> <+> <id, 3> <*> <num, 60>
  2. Syntax Analysis (Parser): Imposes grammar rules. Groups tokens into a hierarchical structure because multiplication has higher precedence than addition. 
    = (assignment)
 ├── id1 (position)
 └── + (addition)
      ├── id2 (initial)
      └── * (multiplication)
           ├── id3 (rate)
           └── num (60)
  3. Semantic Analysis: Verifies semantic consistency. If id3 (rate) is a float, the compiler must convert the integer 60 to float.
    Output: Parse tree with integer 60 explicitly cast to inttofloat(60).
  4. Intermediate Code Generation: Generates three-address code (TAC) instructions: 
    t1 = inttofloat(60)
t2 = id3 * t1
t3 = id2 + t2
id1 = t3
  5. Code Optimization: Constant folds inttofloat(60) into a float constant 60.0 directly at compile time: 
    t1 = id3 * 60.0
id1 = id2 + t1
  6. Target Code Generation: Maps operands to CPU registers and generates assembly: 
    LDF R2, id3         ; Load rate into float register R2
MULF R2, R2, #60.0   ; R2 = R2 * 60.0
LDF R1, id2         ; Load initial into float register R1
ADDF R1, R1, R2      ; R1 = R1 + R2
STF id1, R1         ; Store result to position
Lexical Analysis: Tokens, Patterns, and Lexemes

The first phase of the compiler reads the input character-by-character and classifies text segments:

Concept Definition Example Case
Token Abstract category representing syntactic terminal symbols. id (identifier), num, keyword, relation
Pattern The regular expression rule describing strings belonging to a token. [a-zA-Z_][a-zA-Z0-9_]*
Lexeme Concrete string sequence matched by a pattern in the source. position, rate, 60, if

Lexical Errors vs. Syntax Errors:

Feature Lexical Error Syntax Error
Detection Phase Lexical Analyzer (Scanner). Syntax Analyzer (Parser).
Cause Illegal characters or bad token spelling. Valid tokens placed in structurally invalid combinations.
Example int x = 10#2; (# is not a valid operator). int x = * 5; (illegal operator placement).

Lexical Error Recovery Strategies:

  • Panic Mode Recovery — Deletes successive invalid characters until the scanner reaches a recognizable token boundary.
  • Character Modifications — Tries to insert a missing character, delete a redundant character, transpose adjacent characters, or substitute an invalid character to fix spelling.
Input Buffering: Two-Buffer Scheme & Sentinels

Because filesystem disk reads are slow, reading one character at a time directly from disk blocks compiler performance. The compiler uses a buffer memory scheme:

How It Works:

  • Two buffers, each of size $N$ (typically $4$ KB), are allocated in memory.
  • lexemeBegin Pointer marks the start of the current candidate lexeme.
  • forward Pointer scans forward to inspect characters. When it hits the end of Buffer 1, Buffer 2 is reloaded with fresh input, and the pointer wraps around.

Sentinel Optimization:

Normally, scanning a character requires testing two conditions: (1) did we hit the end of the buffer? and (2) what is the character? To eliminate the boundary test on every single character, we place an EOF sentinel character at the end of each buffer. The boundary reload check is only executed when an EOF is encountered: 

char c = *forward++;
if (c == EOF) {
    if (reached_end_of_buffer1()) {
        reload_buffer2();
        forward = start_of_buffer2;
    } else if (reached_end_of_buffer2()) {
        reload_buffer1();
        forward = start_of_buffer1;
    } else {
        // True End-of-File reached
        terminate_scan();
    }
} else {
    process_char(c);
}
Specification of Tokens (Regular Expressions)

Tokens are mathematically specified using formal regular expressions.

Languages Operations:

  • Union ($L \cup D$) — Strings in $L$ or $D$: {s | s ∈ L or s ∈ D}
  • Concatenation ($LD$) — Paring strings from $L$ and $D$: {xy | x ∈ L and y ∈ D}
  • Kleene Closure ($L^*$) — Zero or more occurrences of language $L$: L* = ∪ L^i (includes ε)
  • Positive Closure ($L^+$) — One or more occurrences of language $L$: L+ = L L* (excludes ε)

Regular Definitions:

Names representing complex regex patterns: 

letter_     → [A-Za-z_]
digit       → [0-9]
identifier  → letter_ ( letter_ | digit )*
digits      → digit+
number      → digits ( . digits )? ( [eE] [+-]? digits )?
Recognition of Tokens & State Diagrams

Regular expressions are compiled into transition state diagrams which are executed as Finite Automata.

Accepting states with a star * signify lookahead retraction, meaning the scanner read one character too far to verify token boundaries (e.g., matching the space after an identifier) and must move the forward pointer back by one position.

LEX Tool Architecture & Execution

LEX is a compiler utility that converts lexical specifications (regular expressions with action C-code) into C scanner source.

Structure of a LEX file (.l):

%{
/* 1. Declarations: imports, variables, header files */
#include <stdio.h>
%}
digit   [0-9]
%%
/* 2. Rules: pattern { C Action Code } */
"if"            { printf("Keyword: IF\n"); }
{digit}+        { printf("Num: %s\n", yytext); }
[ \t\n]+        { /* ignore whitespace */ }
%%
/* 3. User Subroutines: C functions */
int yywrap() { return 1; }
int main() { yylex(); return 0; }

Key Global Interfaces in LEX:

  • yylex() — Core scanning function generated by LEX. When executed, starts reading input.
  • yytext — Character array containing the current string matching the regular expression rules.
  • yyleng — Length of the current matched lexeme.
  • yyin — File pointer representing input source. Defaults to stdin.
  • yywrap() — Executed when EOF is reached. Returns 1 to terminate; returns 0 to switch files.

Rule Matching Priority Resolution:

  1. Longest Match Rule: Prefers the pattern that consumes the largest number of characters. (e.g. matching then as an identifier instead of keyword the followed by n).
  2. First Pattern Rule: If lexeme lengths are equal, prefers the pattern specified highest in the .l file rules. (e.g. mapping if as a keyword rather than identifier because the keyword rule is defined first).
Exam Focus & Important Theory (10 Marks)
  • Phases Tracing — Practice tracing statements like a = b + c * 50 showing all intermediate tokens, trees, and TAC.
  • Differentiate Lexical vs Syntax Errors — Know recovery strategies (panic mode, deletion, character transposition) for 5-mark theory answers.
  • Sentinel Buffer Diagram — Draw the dual buffer sliding windows with boundary EOFs to capture sentinel advantages in speed checks.
Interview Q&A Prep

Q: Why does the compiler use an Intermediate Representation (IR) instead of writing direct machine assembly translation?
A: Decouples frontend syntax checking from backend hardware details. If you support M languages and N hardware architectures, IR reduces required systems from M × N compilers to M frontends + N backends (M + N systems). Also enables machine-neutral optimizations to be shared.

Q: What is the difference between a Lexeme, a Token, and a Pattern?
A: A token is an abstract class representing grammatical terminals (e.g., id). A pattern is the regex describing valid structures. A lexeme is the actual character sequence in the source matching that pattern (e.g., val1).

1. Syntax Analysis — Definition & Position

Syntax analysis (parsing) is the second phase of compilation. It takes the token stream from the lexer and checks whether those tokens form a valid sentence of the grammar. If valid it builds a parse tree; if not, it reports a syntax error.

// Key distinction

Lexical analysis asks: is each token individually well-formed?
Syntax analysis asks: are the tokens arranged in a pattern the grammar permits?
int = x + 5; — every token is valid, but placing int as an LHS is a syntax error.

Context-Free Grammar (CFG)

A CFG is a 4-tuple G = (V, T, P, S) where V = non-terminals, T = terminals, P = productions (A → α), S = start symbol. Every valid program is a sentence of L(G).

Grammar Transformations (required before LL parsing):

AmbiguityString has 2+ parse treesRewrite with precedence levelsE→E+E|E*E|id  →  use E,T,F hierarchy
Parse Tree & Derivations

A Parse Tree shows the derivation graphically. Root = Start Symbol, internal nodes = Non-terminals, leaves = Terminals. The frontier (leaves L→R) = the input string.

FIRST & FOLLOW Sets

Foundation for all parsing tables. Compute these before building any LL(1) or SLR table.

  • FIRST rules: FIRST(terminal)={terminal}; if A→ε add ε; for A→X₁X₂…Xk chain through Xi while ε∈FIRST(Xi)
  • FOLLOW rules: $ in FOLLOW(S); A→αBβ: add FIRST(β)−{ε}; if ε∈FIRST(β) or B at end: add FOLLOW(A)
  • ε is NEVER in FOLLOW. $ is NEVER in FIRST. exam trap

  • Polymorphism — function works on multiple types. Ad-hoc (overloading), parametric (generics), subtype.
Storage Allocation Strategies (high priority)
  • Static allocation — memory fixed at compile time. No recursion. e.g. global vars, FORTRAN.
  • Stack allocation — activation records pushed/popped on call/return. Supports recursion. LIFO.
  • Heap allocation — dynamic memory via malloc/new. Flexible but needs GC or manual free.
Activation Record (AR)
  • Contains: return value, actual parameters, control link (caller's AR), access link, saved machine state, local variables, temporaries.
  • Control link — points to caller's AR (dynamic chain).
  • Access link — points to AR of statically enclosing scope (static chain).
Exam tip: Draw the activation record layout and explain stack vs heap with a diagram. Storage allocation appears in almost every paper it shows up in.

Three-Address Code (3AC)
  • At most one operator per instruction. Form: x = y op z
  • Quadruples — (op, arg1, arg2, result). Easy to reorder; preferred in optimising compilers.
  • Triples — (op, arg1, arg2); result implicit as index. Compact but hard to reorder.
  • Indirect triples — list of pointers to triples; reordering easier than triples.
  • Back patching — filling in incomplete jump targets once the destination is known.
Basic Blocks & Flow Graphs
  • Basic block — maximal sequence of instructions with one entry, one exit, no internal jumps.
  • Leaders — first instruction, any branch target, instruction after a branch.
  • Flow graph — directed graph where nodes = basic blocks, edges = control flow.
DAG (Directed Acyclic Graph)
  • Represents computations in a basic block. Leaves = inputs; interior nodes = operators.
  • Detects common sub-expressions automatically (shared nodes).
  • Algorithm: for each instruction x=y op z, find/create nodes for y,z; create node for op; label with x.
  • Applications: CSE elimination, dead code removal, algebraic simplification.
Register Allocation & Peephole
  • Register allocation — assign variables to registers; use graph colouring (interference graph).
  • Peephole optimisation — small sliding window over code; eliminates redundant loads/stores, unreachable code, jumps to jumps.
Exam tip: Practice DAG construction and 3AC generation by hand. Draw the flow graph for while/for loops step by step.

Code Optimization

Techniques to improve generated code without changing its meaning — mostly theoretical, high repetition in exams.

Sources of Optimization (most repeated topic)
  • Common Sub-expression Elimination (CSE) — avoid recomputing the same expression.
  • Copy Propagation — replace copies (x=y) with the original variable y.
  • Dead Code Elimination — remove code whose result is never used.
  • Constant Folding — evaluate constant expressions at compile time. e.g. 3+4 → 7
  • Strength Reduction — replace expensive op with cheaper one. e.g. x*2 → x+x or x<<1
Loop Optimisations
  • Loop-invariant code motion — move computations that don't change across iterations outside the loop.
  • Induction variable elimination — simplify variables that change by a fixed amount each iteration.
  • Loop unrolling — replicate loop body to reduce branch overhead.
Three Areas of Optimisation
  • Local — within a single basic block. e.g. peephole, CSE in block.
  • Global — across basic blocks in a function. Uses data flow analysis.
  • Inter-procedural — across function boundaries. Most expensive.
Global Data Flow Analysis
  • Computes properties of the program at each point using data flow equations.
  • Reaching definitions — which assignments can reach a given point.
  • Live variable analysis — which variables will be used after this point.
  • Solved iteratively (forward or backward) until a fixed point is reached.
Exam tip: Learn the names and one-line definitions of all sources of optimisation — they appear as "write a short note on" questions almost every year.
